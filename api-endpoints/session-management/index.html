    Session Management

    Session management endpoints control the streaming session lifecycle from creation through termination. These endpoints support different session types (TLC, Broker, Monitor) and protocol configurations (singleplex and multiplex).

    Note: In this API, "TLC" refers to any data-producing entity in the C-ITS ecosystem. This includes traffic controllers, sensors, detection systems, and other connected devices. The term "Subject" is used interchangeably with "TLC" to reflect the broader scope of entities that can participate in data streaming sessions.

    Overview

    Session management allows you to:

    • Create new streaming sessions for data exchange
    • List and retrieve active sessions you have access to
    • Update session configurations (multiplex sessions only)
    • Terminate active sessions

    Results are automatically filtered based on the domain associated with your API key's authorization.

    Available Operations

    List Active Sessions

    GET /v1/sessions

    Retrieve active sessions with automatic filtering based on your API key's authorization context.

    Required Roles: TLC Admin/System, Broker Admin/System, Monitor Admin/System

    Optional Query Parameters:

    • type - Filter by session type (TLC, Broker, Monitor)
    • protocol - Filter by protocol (TCPStreaming_Singleplex, TCPStreaming_Multiplex)

    Response: Array of session objects with domain, type, token, protocol, and connection details.

    Create Session

    POST /v1/sessions

    Create a new streaming session within your authorized domain context.

    Required Roles: TLC Admin/System, Broker Admin/System, Monitor Admin/System

    Request Body: 

    {
  "domain": "your-domain",
  "type": "TLC",
  "protocol": "TCPStreaming_Singleplex",
  "details": {
    "securityMode": "NONE",
    "tlcIdentifier": "tlc12345"
  }
}

    Key Points:

    • Domain in the request must match your API key's authorized domain
    • Session types: TLC (data producers), Broker (distributors), Monitor (observers)
    • Protocols: TCPStreaming_Singleplex (single TLC) or TCPStreaming_Multiplex (multiple TLCs)
    • Security modes: NONE (unencrypted) or TLSv1.2 (encrypted)

    Retrieve Specific Session

    GET /v1/sessions/{token}

    Get detailed information about a specific session using its token.

    Required Roles: TLC Admin/System, Broker Admin/System, Monitor Admin/System

    Response: Single session object with complete connection details and configuration.

    Update Session

    PUT /v1/sessions/{token}

    Update session configuration (TCP Streaming Multiplex sessions only).

    Required Roles: TLC Admin/System, Broker Admin/System, Monitor Admin/System

    Request Body: 

    {
  "securityMode": "TLSv1.2",
  "tlcIdentifiers": ["tlc12345", "tlc67890"]
}

    Important:

    • Only multiplex sessions support updates
    • The securityMode field is mandatory but cannot be changed - it must match the existing session's security mode
    • Updates can modify the TLC scope by changing tlcIdentifiers

    End Session

    DELETE /v1/sessions/{token}

    Terminate an active session immediately.

    Required Roles: TLC Admin, Broker Admin, Monitor Admin

    Response: 204 No Content on successful termination.

    Important: Session termination is immediate and irreversible. Any active TCP connections will be closed.

    Session Types

    TLC Sessions

    TLC sessions are for TLCs (data-producing entities) that own and generate data:

    • Purpose: Data production and publishing
    • Data Flow: Outbound (publishing data to other participants)
    • Access Pattern: Write-focused with optional read access
    • Use Cases: Traffic controllers, sensors, detection systems, and other connected TLCs

    Broker Sessions

    Broker sessions handle data distribution and routing:

    • Purpose: Data relay and distribution
    • Data Flow: Bidirectional (receiving and redistributing data)
    • Access Pattern: Read-write with routing logic
    • Use Cases: Data aggregators, cloud connectors, central systems

    Monitor Sessions

    Monitor sessions provide read-only access for observation:

    • Purpose: Data monitoring and analytics
    • Data Flow: Inbound only (consuming data)
    • Access Pattern: Read-only access
    • Use Cases: Analytics systems, dashboards, logging services

    Protocol Types

    TCP Streaming Singleplex

    Single-participant streaming with one TLC identifier:

    {
  "domain": "your-domain",
  "type": "TLC",
  "protocol": "TCPStreaming_Singleplex",
  "details": {
    "securityMode": "NONE",
    "tlcIdentifier": "tlc12345"
  }
}

    Characteristics:

    • One TLC identifier per session
    • Simpler session management
    • Direct TLC-to-consumer communication
    • Lower overhead for single-source scenarios

    TCP Streaming Multiplex

    Multi-participant streaming with multiple TLC identifiers:

    {
  "domain": "your-domain", 
  "type": "Broker",
  "protocol": "TCPStreaming_Multiplex",
  "details": {
    "securityMode": "TLSv1.2",
    "tlcIdentifiers": ["tlc12345", "tlc67890", "tlc11111"]
  }
}

    Characteristics:

    • Multiple TLC identifiers per session
    • Dynamic TLC scope updates supported
    • Aggregated data streams
    • Higher efficiency for multi-source scenarios

    Security Modes

    NONE

    Unencrypted TCP streaming:

    • Performance: Higher throughput, lower latency
    • Security: No encryption, data transmitted in plaintext
    • Use Cases: Internal networks, development environments
    • Considerations: Not recommended for production over untrusted networks

    TLSv1.2

    Encrypted TCP streaming using TLS 1.2:

    • Performance: Lower throughput due to encryption overhead
    • Security: Industry-standard encryption protection
    • Use Cases: Production environments, cross-network communication
    • Considerations: Requires TLS configuration

    Session Lifecycle

    Creation Process

    1. Ensure referenced TLCs (subjects) are registered in your domain
    2. Choose appropriate session type based on data flow requirements
    3. Select protocol based on TLC scope needs
    4. Configure security mode based on network environment
    5. Create session and receive connection details
    6. Establish TCP connection using provided host and port
    7. Begin data streaming according to protocol specifications

    Connection Details

    Successful session creation returns connection information:

    {
  "domain": "your-domain",
  "type": "TLC",
  "token": "session-token-123",
  "protocol": "TCPStreaming_Singleplex",
  "details": {
    "securityMode": "NONE",
    "listener": {
      "host": "streaming.example.com",
      "port": 8443,
      "expiration": "2024-01-15T10:30:00Z"
    },
    "tlcIdentifier": "tlc12345"
  }
}

    Active Session Management

    While sessions are active:

    • Monitor connection health using keep-alive mechanisms
    • Handle rate limiting and throughput constraints
    • Configure TLS for secure sessions
    • Update multiplex sessions as TLC scope changes
    • Monitor session logs for troubleshooting

    Termination

    Sessions end through:

    • Explicit termination: DELETE request to end session
    • Automatic expiration: Sessions expire based on configuration
    • Connection failure: Network or protocol errors
    • System maintenance: Planned or unplanned service interruptions

    Error Responses

    Common error scenarios:

    • 400 Bad Request - Invalid session configuration or unknown TLC identifiers
    • 401 Unauthorized - Invalid or missing authorization token
    • 403 Forbidden - Insufficient permissions or domain mismatch
    • 404 Not Found - Session token not found or not accessible

    Best Practices

    Session Planning

    • Choose session types based on actual data flow patterns
    • Use singleplex for simple point-to-point scenarios
    • Use multiplex for aggregation and distribution scenarios
    • Plan security mode based on network environment

    Performance Optimization

    • Monitor session throughput and adjust as needed
    • Use appropriate security modes for performance requirements
    • Consider connection pooling for high-frequency operations
    • Implement proper error handling and retry logic

    Security Considerations

    • Use TLS 1.2 for production environments
    • Regularly rotate session tokens through re-creation
    • Monitor session access patterns for anomalies
    • Implement proper authentication and authorization
